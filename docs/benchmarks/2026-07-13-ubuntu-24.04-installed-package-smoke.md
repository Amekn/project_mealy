## Generated systemd user-service integration

A later privileged Ubuntu 24.04 PID-1 container exercised the generated unit itself through a real
systemd user manager. This was intentionally deeper than the earlier package smoke, which checked
the unit text but ran the daemon directly. The first exact unit launch exposed root-owned system
files as overflow UID 65534 inside systemd's single-identity user namespace, so the exact trusted
`/usr/bin/ldd` check initially failed. The replacement trust check accepts that representation only
when the effective UID/GID maps are exact one-entry identity maps and the entire same-owner path
chain is non-writable and mounted read-only.

The next probes isolated `ProtectHostname=true`, `ProtectKernelLogs=true`, and
`ProtectKernelTunables=true` as incompatible with Bubblewrap's nested hostname and fresh `/proc`
setup. Replacing their process-view intent with `ProtectProc=invisible` and `ProcSubset=pid` made
both sandbox profiles enforceable. A real approved write then still failed with `path_denied`:
systemd's `RestrictSUIDSGID=true` seccomp policy blocks the worker's secure
`openat2(O_CREAT, ...)` call. Removing only that incompatible restriction while retaining
`NoNewPrivileges`, the private umask, Bubblewrap capability dropping, read-only system/home views,
and the other unit restrictions made the exact effect succeed.

This historical container result was subsequently superseded for release qualification by the
GitHub-hosted Ubuntu 24.04 user-manager test. The ordinary host keeps AppArmor's restriction on
unprivileged user namespaces, so systemd-native namespace/capability directives fail before daemon
exec. GitHub run `29357794460` then showed that moving the outer boundary into trusted
`/usr/bin/bwrap` also fails: Ubuntu's packaged child profile removes capabilities needed by the
nested per-tool Bubblewrap. The final candidate directly executes the daemon under
rootless-compatible systemd process/cgroup controls, and the checked host smoke must prove that the
governed per-request mutation remains sandboxed and functional.

The resulting checked `scripts/systemd-service-smoke.sh` repeated the complete generated-unit
